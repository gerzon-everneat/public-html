User Story and Behavioural Reference

Admin booking lifecycle, creation through payment collection

An end-to-end account of what an Owner or Admin can do in the Neatr admin portal: every wizard option, what selecting it changes, how the booking moves through its states, who may move it, and what the Booking Summary shows on each tab. Written against the source, so it records what the system actually does, including the places where the UI promises something it does not deliver.

Preconditions: what must exist first

As an admin

I want the catalog configured before I take a booking, so the wizard has services, pricing and a service area to work with.

There is no seeded catalog

Categories, services, extras, spaces and frequencies are entirely tenant-authored at runtime. A brand-new tenant starts completely empty, and the public catalog returns explicit no_categories and no_services states, so the business shows as "not taking bookings yet". Nothing in the product ships a default service list.

The only data seeded automatically is three Office attributes (Type, Layout, Foot traffic), bound to categories named exactly Office or Commercial.

  • An empty tenant produces a wizard that cannot reach a quote. This is by design, not a fault.
  • Service-area enforcement is conditional. If the tenant has authored no zip groups at all, any zip passes. The gate only bites once a service area exists.

Category branching: the three shapes of a booking

The wizard forks on two booleans derived from the category name the tenant typed, matched case-insensitively as a substring:

isOffice    = categoryName contains "office" OR contains "commercial"
isAirbnb    = categoryName contains "airbnb"
residential = neither of the above (the implicit else branch)
Substring matching on free text, so treat it as fragile

A category named "Post-Construction Commercial" silently becomes an office booking, with office property, scope and payment steps. Renaming "Airbnb Turnover" to "Short-term rental" silently drops all Airbnb behaviour, and the dual check-in and check-out schedule simply stops appearing.

There is a second, sharper trap. The backend attribute seeder matches Office and Commercial exactly, while the frontend uses the substring regex. A category named "Office Cleaning" therefore gets the full office UI but never receives the three default attributes, so the property step renders with nothing in it.

No step is added or removed by category. Only the component, field visibility and validation change. Step count varies only through the tenant's wizard config.

Changing category is destructive

Picking a different category in Step 2 wipes scope of work, quote, protocol, both Airbnb protocols and the schedule back to initial state. The behaviour is intentional, since the downstream data is category-shaped, but it is silent. An admin who changes category late loses everything after Step 2 with no confirmation prompt.

The wizard: every step, every option

Entry point /home/bookings/new, which also supports ?editId=, ?resume= and ?duplicateFrom=.

Step order is not fixed

The order below is the default. Steps come from a tenant-persisted wizard config, so each can be disabled or reordered, and tenants may insert custom steps. The final Review step is always appended and cannot be configured away.

Navigation is hierarchically invalidated. Changing the customer, the qualification, the scope, or a price-derivation field revokes forward navigation and drops the admin back to that step, discarding progress made after it.

Fixed-dollar frequency discounts render two different totals

The Quote step honours a frequency discount of type "amount". The context panel, confirmation step and quote email use a simpler percent-only formula that ignores it. A tenant using fixed-dollar recurring discounts will see the quote disagree with the confirmation and with the email the customer receives. Percent discounts are unaffected.

  • Expectations
  • Send Quote offers Email or Phone, disabled when that contact detail is absent, with a 60 second resend cooldown. Editing after sending switches the button to Send Updated Quote.
  • Declined saves the booking as quote with a declined marker, then closes.
  • Next marks the quote accepted.
  • Office bookings hide Send Quote and Declined entirely.
  • A tenant frequency row renamed away from a recognised spelling falls back to one-time pricing silently, a documented past bug where "Bi-Weekly" once matched nothing.

Protocol

protocol
As an admin

I want to tune the checklist the provider will execute on site.

Seeded from the selected service's checklist space types. Per space the admin can expand, star, rename, set estimated time, add a note, check tasks, search and add tasks, or add a whole space. A free-text service note covers the entire job.

  • Expectations
  • Changing Service Level in Step 4 reseeds this checklist.
  • No validation runs, so this step never blocks Next even with an empty checklist.
  • What is authored here becomes the provider's completion gate. A provider cannot mark the job complete with tasks outstanding, while Owner and Admin are deliberately exempt.

How availability is computed

The server returns skill and service-area disqualifiers. The client then re-derives status against the chosen time, all in business-timezone wall minutes, never device-local:

  1. no_schedule and skill_mismatch are static and survive any time change.
  2. Approved time off on that date is treated as booked.
  3. The whole job must fit inside working hours.
  4. No overlap with existing bookings, and arrival-window jobs block a wider band.
  • Expectations
  • Nothing blocks the admin. Booked, unscheduled and skill-mismatched providers all remain clickable, because force-assign is intentional.
  • Assigning a provider on approved time off raises a confirm dialog offering Cancel or Confirm Anyway.
  • A provider with no working hours shows a deep link to set up their schedule.
  • Provider is optional. With none picked, the button reads "Continue (unassigned)" and the booking can be assigned later from the drawer.
  • Admin auto-assign never overwrites an existing selection.

Two office behaviours worth knowing

On the office branch a card is saved, never charged at booking time, because collection happens later against the invoice. And regardless of what the UI shows selected, office bookings persist invoice as the payment option. The displayed value and the stored value differ by design.

Office adjustments do not feed the pricing engine. They are snapshot only, so an adjustment entered here will not change the computed total.

  • Expectations
  • The step prefills from the customer's payment profile: an invoice preference gives invoice, ACH with a bank name gives ACH, otherwise an existing card gives existing card.
  • If the stored preference is no longer an enabled method, it falls back to the first available one and writes that back, so the guards match what is on screen.
  • Picking a saved card also sets it as that customer's default, and a failure there is non-fatal.
  • With no saved cards the step reads "No saved cards for this customer." and offers an Add card action.

Review and confirm

always appended

A read-only summary of service, spaces, add-ons with quantities, date and time, frequency, address, access and customer, plus:

  • Send Confirmation. An Email checkbox, disabled without an email, and an SMS checkbox, disabled without a phone. Both are forced off and disabled when "Exclude customer notification" was ticked on the Quote step.
  • Charges. The full breakdown plus payment method, totalled as "Due today" or, for office, "Total" with an invoice-after-service note.
  • Internal Notes, admin only.

What "Confirm" actually does

  1. Business timezone must be loaded, otherwise "Still loading the business's timezone".
  2. New credit card selected but not saved is blocked.
  3. Existing credit card selected with no card chosen is blocked.
  4. Missing date or start time is blocked.
  5. A re-entrancy guard prevents a double-click from creating two bookings.

Drafts happen whether you ask for them or not

Before a service is picked, a debounced session draft is kept in the browser and offered back as "Resume previous session?". Once a service exists, every Next writes a real draft booking to the database. Abandoned wizards therefore leave draft rows behind. They are hidden from the default list view, but they do exist.

Explicit saves are also available: Save as Draft before the quote step, and Save as Quote from the quote step onward.

The state machine

draft       -> quote, pending, confirmed, cancelled
quote       -> confirmed, cancelled
pending     -> confirmed, cancelled
confirmed   -> assigned, on_the_way, cancelled, rescheduled
assigned    -> on_the_way, in_progress, cancelled, rescheduled, confirmed
on_the_way  -> in_progress, cancelled
in_progress -> completed, cancelled
completed   -> (terminal)
cancelled   -> rescheduled
rescheduled -> confirmed, rescheduled, cancelled

completed is the only terminal state. Note that assigned back to confirmed is how a booking gets unassigned, and that cancelled to rescheduled means a cancelled booking can be revived.

Arrival GPS never blocks anything

Provider arrival records latitude, longitude, accuracy and distance as an audit trail only. Proximity is enforced client side, and the server deliberately accepts a bad or missing fix, because GPS is unreliable indoors and a legitimate arrival must never be blocked. Arrival is also not a precondition for starting the job.

Provider assignment

As an admin

I want to put gerzon.ustp@gmail.com on this job, during creation or afterwards from the list or the drawer.

There is no dedicated assign endpoint. Three paths write the provider:

PathHow the admin reaches itNotes
Status change to assignedDrawer Schedule tab, or the list row action Assign providerRuns every guard, including availability
Generic editDrawer reassign, or Edit BookingChanges providers without a status change
Born assignedWizard Step 7Provider set at creation
Preconditions on assign
  • At least one provider must be supplied, or already on the booking.
  • Availability is checked only for future-dated bookings. An active availability row for that weekday must exist, otherwise the request fails with 422.
  • No skill check is enforced at assign time. Skills only rank the suggestion list, so a provider with no matching expertise can still be assigned.
  • No double-booking check. Conflict detection is a UI affordance, not a server guard.
  • Providers cannot self-assign.
Side effects of assigning
  • A real-time status broadcast to the tenant, plus a personal update to each affected provider.
  • The availability cache is invalidated.
  • A booking.assigned notification goes to every provider on the booking.
  • The booking's chat thread swaps the old provider for the new one.
  • Unassigning clears the provider fields and removes them from the chat.
  • Expectations
  • A suggested-pros helper ranks providers whose expertise covers the service, falling back to anyone with skills on file. Owner and Admin only.
  • Removing the last provider from the drawer's Schedule tab downgrades the booking to confirmed.

Payment collection

There are two independent payment fields, and this is the most misread part of the system

paymentStatus tracks card money, moving through none, pending, then succeeded, failed or refunded.

invoicePaymentStatus tracks invoice settlement, moving through uninitiated, pending, then paid or declined.

Cash, ACH and manual collection move only the invoice field. paymentStatus stays none forever. A fully collected office booking will still read as unpaid on any card-centric view, including the Billing tab's chip, which derives from paymentStatus. Reconcile on the right field.

Charging a specific saved card is ownership-checked

When the collection modal picks a particular card, the server verifies that the card belongs to this customer before charging it, and rejects anything else. Multi-card customers are safe from cross-charging.

Upfront charge at confirmation

If the tenant has "collect payment at booking" enabled, confirming a card booking charges immediately. A failed charge aborts the confirm, so the booking does not move to confirmed carrying an unpaid balance.

Automatic reconciliation

A scheduled job re-queries the processor for bookings stuck in pending for more than ten minutes, so a dropped webhook self-heals instead of stranding the payment.

  • Expectations for the happy path
  • Residential card booking: confirm, then paymentStatus succeeded, and the Billing tab shows Paid.
  • Office invoice booking: confirm, card stored and nothing charged, invoice sent, then invoicePaymentStatus paid while paymentStatus remains none.

The drawer's status chip is not the one from the list

The list chip colours every real status correctly. The detail chip only maps working, upcoming and complete, none of which are real booking statuses, so in the drawer every booking renders grey. Read status from the list, or from which action buttons are available, rather than from the drawer chip's colour.

Refunds and card charges are not in this drawer

Refund, payment collection, add-tip and mark-as-paid modals live in Finances and Payments. The lifecycle from booking to money crosses two screens, because the drawer's Billing tab reports state rather than collecting it.

Gating, in order

  1. Tenant v2 flag. Off means legacy behaviour with all channels on, and lookup failures fail open.
  2. Channel matrix, per event and audience.
  3. SMS consent. Customer SMS requires the booking's consent box to have been ticked. Business audiences are exempt.
  4. Per-user preferences, asymmetric by design: customers are opt-out, so silence means send, while providers and admins are opt-in, so silence means do not send.
  5. Destination, normalised to E.164, falling back to the booking phone for customers and the profile phone for staff.
Why a provider "never gets SMS"

Because business-audience SMS is opt-in. A provider with no explicit per-event SMS preference receives nothing, and that looks identical to a delivery failure. Check the preference before investigating the carrier.

Caveats for this exact scenario

Consequences of the specific customer, provider and address used in this story.

A Cagayan de Oro address will fight the service-area gate

Step 2 validates a five digit zip against the tenant's authored zip groups. Philippine postal codes are four digits, and Cagayan de Oro is 9000. Unless the tenant has explicitly authored that code into a zip group, the step shows Outside your service area and Next stays blocked, so the wizard cannot proceed.

There is one escape hatch: a tenant with no zip groups authored at all skips the check entirely. This address therefore works on a fresh tenant and fails on a configured one, which makes it a poor smoke-test address, since the failure depends on tenant config rather than on the code path under test.

SMS to a Philippine number will not deliver

The messaging account is provisioned for a US A2P campaign with geographic permissions restricted to the US, and non-US destinations are rejected by the carrier. Every SMS leg of this lifecycle, including confirmation, on-the-way, reminders and provider assignment, will fail for a Philippine number even though the app reports the send as attempted.

Verify these legs by email, or use US numbers for both gerzon.informatics@gmail.com and gerzon.ustp@gmail.com.

The provider needs an availability row for the weekday

Assigning gerzon.ustp@gmail.com to a future booking requires an active availability row for that day of the week, or the API refuses with "Provider is not available on this day". Same-day bookings skip the check. If assignment fails, set up the provider's availability first. The wizard will still let you select them, so the failure surfaces late.

Timezone

All scheduling is anchored to the business profile timezone, not the browser's. Driving this scenario from a Philippine browser against a US-timezone tenant will show times that look shifted, which is correct behaviour rather than a bug. The list shows business time with an optional "your time" secondary line.
