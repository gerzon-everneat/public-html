ACH & Bank Transfer: Payment Step Audit

spec verified 2026-07-21

The admin booking wizard has a payment step for both residential and office/commercial bookings. This section audits the gap between what the UI implies and what is actually implemented in the codebase.

The two models (conflated in the UI)

Bank Transfer (push)ACH Debit (pull)
DirectionCustomer sends money to tenantTenant takes money from customer
Processor neededNoneStripe + Connect
Mandate requiredNoYes (Nacha WEB debit)
Return liabilityNoneUp to 60 calendar days, no appeal
Current statusUI shows it, reconciled manually out-of-bandNot built. Phase B is blocked on a business decision

Verified codebase gaps

CriticalNo tenant receiving account is stored anywhere. The UI lets admins select "Bank transfer" as a payment method, but there is no remittanceInstructions field in any settings model. Customers receive no account or routing details; payment is reconciled by hand via a 4-digit memo out-of-band.
HighOffice booking payment method list ignores tenant toggles. OfficePaymentStep.tsx:28 renders a hardcoded PAYMENT_METHODS const with no showForTenant filter. Disabling ACH in Operations rules hides it from residential bookings but has no effect on office ones.
Highprofile.achPaymentMethodId is a dead field. Defined at profile.model.ts:72, read in 3 places, written in 0. The field is reserved for a Stripe Financial Connections integration that has not been built.
HighPayment state enum is missing a "returned" state. paymentStatus supports none | pending | succeeded | failed | refunded. An ACH return (R10) arrives after succeeded and stays silently marked paid; there is no enum value to express "was paid, then clawed back".
MediumLabels are misleading. The "ACH" option in the office wizard has never been a Stripe debit (it is a push/bank-transfer); calling it ACH sets incorrect expectations for admins and customers, and will conflict with the name once real ACH debit is built.
MediumNo backend guard on the office payment path. The two server-side card guards in booking.ts key off selectedPaymentOption / preferredPaymentMethod, both of which office bookings force to "invoice". A direct API call can confirm an office card booking with no card on file; the wizard is the only gate.
MediumpaymentStepGate.ts is implemented but not yet landed. Validation logic with 18 tests exists uncommitted. Card requires paymentMethodId; ACH requires bank name and 4-digit last4; direct deposit requires nothing. This should be committed.
LowMercury integration is unused. mercury.service.ts exists in the codebase but is never called (confirmed unused by product owner 2026-07-21). Dead code that adds surface area for confusion.

User stories: what the payment step should support

Office / Commercial
  • PAY-O1 As an admin, I can set a billing cycle (Per Clean or Monthly) and settlement method so that AR knows when to invoice and how payment arrives
  • PAY-O2 As an admin, I can attach a saved card for post-service collection so that the invoice can be collected after the job
  • PAY-O3 As an admin, I can record the client's bank name and last 4 digits for bank-transfer reconciliation so that finance can match incoming transfers to the invoice
  • PAY-O4 As an admin, I can select Direct Deposit with no further input so that wire-paying clients are not blocked by data entry
Residential / Airbnb
  • PAY-R1 As an admin, I can select a saved card for a returning customer so that I do not re-key details they already provided
  • PAY-R2 As an admin, I can capture a new card inline so that I can book without leaving the wizard
  • PAY-R3 As an admin, I can record bank transfer details (same rule as PAY-O3) so that reconciliation works identically across categories
  • PAY-R4 As an admin, I can select Invoice and move on with no payment instrument so that invoice-terms customers are not blocked

FlowMethodBlocks until

Implementation roadmap (proposed)

Phase 0: CleanupNo dependencies
  • Filter office PAYMENT_METHODS through showForTenant predicate (same as residential)
  • Rename "ACH" option to "Bank Transfer": it has never been a Stripe debit
  • Land paymentStepGate.ts + 18 tests (already written, uncommitted)
Phase A: Manual bank transferNo Stripe required
  • Add remittanceInstructions to business settings, encrypted, tenant-scoped, Owner/Admin only
  • Render remittance details on invoices and booking confirmation emails
  • Surface a booking/invoice reference number in the UI as the transfer memo key
Note Exit criteria: admin books an office job on bank-transfer terms; customer receives an invoice with everything needed to send payment; finance matches the incoming transfer without asking anyone
Phase B: Stripe ACH debit (pull)Blocked
  • Blocked on a business decision: Connect direct charges vs. accepting platform-level ACH return liability (up to 60 days, no appeal)
  • Requires Stripe Financial Connections for mandate capture; admin cannot enroll a customer's bank account on their behalf
  • Nacha fraud monitoring Phase 1 (Mar 2026) and Phase 2 (Jun 2026) are already in force
  • Economics: 0.8% capped at $5 vs. card at 2.9% + 30¢; the cap only pays off on commercial invoice-terms accounts above ~$625
Critical Do not start Phase B until the Connect liability decision is made in writing
